**Step 3 — Refactor `NavRoleGuard` in `App.tsx` (lines 36-52):**

Currently:
```tsx
function NavRoleGuard({ allowedRoles, children }) {
  const { user } = useAuth()
  if (!allowedRoles.includes(user.role)) return null
  return children
}
```

Replace with:
```tsx
/**
 * @description Route guard that hides children unless the current user has the required catalog permission.
 * @param props.requiredPermission - PERMISSION_KEYS entry the user must hold
 * @param props.children - Protected content
 * @returns Children or null
 */
function NavRoleGuard({ requiredPermission, children }: { requiredPermission: PermissionKey; children: React.ReactNode }) {
  const allowed = useHasPermission(requiredPermission)
  if (!allowed) return null
  return children
}
```

Update every call site of `NavRoleGuard` in `App.tsx` to pass `requiredPermission={PERMISSION_KEYS.X}` instead of `allowedRoles={[...]}`. **Use the SAME mapping table from Step 1, indexed by route path.** A `<NavRoleGuard>` wrapping `/iam/users` route uses `PERMISSION_KEYS.USERS_VIEW`; one wrapping `/system/audit-log` uses `PERMISSION_KEYS.AUDIT_VIEW`; etc. If a `NavRoleGuard` call site wraps a route that is not in the Step 1 mapping table, **stop and surface to user** — that's a route the planner didn't anticipate and needs explicit pinning.
