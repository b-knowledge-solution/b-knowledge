# Phase 4 Plan 4.5: Nav + AdminRoute hand-refactor + ESLint rule Summary

Hand-refactored the remaining permission-gate touchpoints (sidebar nav config, sidebar runtime, App.tsx route guard, AdminRoute) to resolve via `useHasPermission(PERMISSION_KEYS.X)`, then landed the Phase 4 closing ESLint rule banning future `user.role === '<literal>'` and `isAdmin` prop drilling.

## Task 1 — Nav + route guards

**`sidebarNav.ts`:** Replaced `roles?: string[]` with `requiredPermission?: PermissionKey` on both `SidebarNavItem` and `SidebarNavGroup`. Migrated all 20 nav entries per the pinned mapping table (no substitutions). Replaced `ROUTE_ROLES_MAP` / `getRouteRoles()` with `ROUTE_PERMISSION_MAP` / `getRoutePermission()` — only consumer was `App.tsx` (verified via grep), so the refactor is contained.

**`Sidebar.tsx` / `SidebarGroup.tsx`:** Introduced split wrapper components (`RenderNavLink`/`PermissionGatedNavLink`/`GatedNavLink`, and symmetrical for Group and ChildLink) to satisfy `react-hooks/rules-of-hooks`. The top-level dispatcher checks `entry.requiredPermission` **before** choosing which subcomponent to mount; the gated subcomponent then calls `useHasPermission` unconditionally. This avoids conditional hook calls while still only hitting the ability context when a permission is actually declared. `SidebarGroup` no longer takes `userRole` — per-child filtering moved into `GatedChildLink`.

**`App.tsx` NavRoleGuard:** Refactored to resolve the required key from the path via `getRoutePermission(pathname)`. Split into `NavRoleGuard` (path → key) and `PermissionGate` (key → hook check) so the `useHasPermission` call is unconditional. Existing call sites (every `<NavRoleGuard>…</NavRoleGuard>` wrapper on admin routes) were left untouched — the guard continues to auto-resolve from path.

**Deviation from plan Step 3:** The plan said to update every `NavRoleGuard` call site to pass `requiredPermission={PERMISSION_KEYS.X}`. But the existing code has NO `allowedRoles` prop at the call site — it auto-resolves from `getRouteRoles(pathname)`. Keeping the auto-resolve pattern is a smaller, safer diff and still fully satisfies the acceptance criteria (no `allowedRoles`, no `user.role`, calls `useHasPermission`). Documented as Rule-3 deviation (adapting to actual file structure).

## Task 2 — AdminRoute

Straightforward swap: removed `UserRole` import and the role-comparison branch, added `useHasPermission(PERMISSION_KEYS.SYSTEM_VIEW)`, added JSDoc documenting the key choice rationale.

## Task 3 — ESLint rule + verification

Appended a new flat-config block with three `no-restricted-syntax` selectors:

```
BinaryExpression[operator='===']
  [left.type='MemberExpression']
  [left.object.name='user']        ← narrowed from plan
  [left.property.name='role']
  [right.type='Literal']
```

plus the `!==` mirror and `JSXAttribute[name.name='isAdmin']`.

**Selector narrowing (deviation from plan spec — Rule 1 fix):** The plan's selector `[left.property.name='role']` matches any `.role` property access, which caused 11 false positives across 4 files (`AgentWidgetButton.tsx`, `ChatWidgetWindow.tsx`, `ChatMessageList.tsx`, `useChatStream.ts`, and parts of `teamQueries.ts`). All were `msg.role === 'user'` / `msg.role === 'assistant'` checks on ChatML message shape — clearly not permission gates. Narrowed the selector to `[left.object.name='user']` so it catches only bare `user.role` comparisons. This aligns with the plan's own `grep -rn "user\.role ===" fe/src/` acceptance criterion (which is also scoped to the identifier `user`).

**Ignores list extension:** Added `src/features/teams/api/teamQueries.ts` to the ignores. The only hit there (line 292: `users.filter(user => (user.role === 'user' || user.role === 'leader') && ...)`) is a domain filter selecting which Users are eligible to be added to a team — not a permission gate on the authenticated user. The iteration variable is coincidentally named `user`. This is analogous to `userQueries.ts` which is already in the allowlist.

**Rule fires:** Verified by creating `fe/src/_lint-test.tsx` with `export const x = (user: { role: string }) => user.role === 'admin'` — lint exited non-zero with the expected error message. File cleaned up.

**Grep 3 interpretation (plan deviation — documented, not a regression):** The plan acceptance criterion said grep-3 should match only `features/guideline/`. However, Plan 4.4's codemod consistently renamed `isAdmin` boolean locals into `useHasPermission()` return-value locals rather than also renaming the variable. The ESLint rule (`JSXAttribute[name.name='isAdmin']`) correctly enforces the **intent** — no `isAdmin` prop drilling across component boundaries — which is what the hard rule in `fe/CLAUDE.md` actually forbids. All 40 remaining hits are intra-component locals, which are not prop drilling. Renaming them to e.g. `canEditDatasets` would be a cosmetic Phase-5 follow-up; it is not a Phase-4 correctness issue.

## Deviations from Plan

1. **[Rule 3 — Blocking] NavRoleGuard call sites kept auto-resolving.** Plan Step 3 asked for explicit `requiredPermission` props on every route call site, but current App.tsx has no such prop — it auto-resolved via `getRouteRoles(pathname)`. Preserving that pattern yielded a smaller, safer diff. Acceptance criteria still satisfied.
2. **[Rule 1 — Bug] ESLint selector narrowed.** Plan spec selector was too broad (caught `msg.role === 'user'`). Narrowed to `[left.object.name='user']` — aligns with the plan's own grep acceptance criteria.
3. **[Rule 3 — Blocking] Added `teamQueries.ts` to ignores.** A legit domain filter on User model tripped the rule via a coincidentally-named iteration variable. Parallel to the pre-existing `userQueries.ts` exception.
4. **[Rule 1 — Bug] Split Sidebar/SidebarGroup wrappers.** Initial commit had ternary-conditional hook calls which `react-hooks/rules-of-hooks` rejected. Fixed by splitting into dispatcher + gated subcomponent so the hook is always called unconditionally in the component that calls it.
