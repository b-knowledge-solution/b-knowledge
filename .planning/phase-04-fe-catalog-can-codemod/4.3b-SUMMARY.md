# Phase 4 Plan 4.3b: Pattern D — Consumer-side isAdmin Prop Elimination Summary

Adds Pattern D to the FE permissions codemod so child components that receive `isAdmin: boolean` as a prop are migrated to compute it locally via `useHasPermission(PERMISSION_KEYS.X)`.

## What Pattern D Does

Fires only when ALL hold:
1. The file is a key in `FILE_KEY_MAP` with a non-null value.
2. The file declares a React component whose first parameter is an ObjectBindingPattern containing `isAdmin`.
3. The body is a Block (so we can insert statements).
4. The body is not already migrated (idempotency: first statement does not already start with `useHasPermission(PERMISSION_KEYS.`).

When it fires it applies four transforms atomically:
- **D.1** strips `isAdmin` from any matching `interface` PropertySignature OR `type` alias TypeLiteral member. Match heuristic: prefer the type-annotation name on the parameter; otherwise touch any interface/alias property named `isAdmin`.
- **D.3** injects `const isAdmin = useHasPermission(PERMISSION_KEYS.<KEY>)` as the first statement of the FIRST candidate's body. (Done before D.2 so that descendant-forgetting from D.2's `replaceWithText` doesn't invalidate the body reference.) If a file has multiple candidates a `// TODO(perm-codemod): multiple isAdmin consumers...` warning line is also prepended.
- **D.2** rebuilds each candidate's destructuring pattern via `replaceWithText`, omitting `isAdmin`. (BindingElement has no `.remove()` in ts-morph, so we round-trip the text.)
- **IMP** reuses the existing import-insertion block to add `useHasPermission` and `PERMISSION_KEYS` imports if absent.

JSX usages of `isAdmin` are intentionally left untouched — they transparently bind to the new local const.

## Pattern Interaction

- **Pattern A** still strips `isAdmin={...}` JSX attributes at parent call sites. Pattern D handles the corresponding child consumer.
- **Pattern B** owns `const isAdmin = user?.role === ...` rewrites; it cannot fire on a file whose `isAdmin` originates as a prop because no such const exists. Pattern D and B are disjoint by construction in the current inventory.
- **Pattern C / C-2** (TODO annotators) are independent and run regardless.

## Idempotency

Pattern D no-ops when the first statement of the first candidate's body already starts with `useHasPermission(PERMISSION_KEYS.`. Verified via fixture 10. Existing fixtures 06 (`already-migrated`) and 07 (`existing-import`) still pass — Pattern D does not fire on them because they have no `isAdmin` in the destructuring pattern.

## Project Manipulation Settings

Added `indentationText: IndentationText.TwoSpaces` to the in-memory ts-morph project so that injected statements match repo style (2-space indent). This was previously defaulting to 4 spaces. The change is invisible to existing fixtures since none of them had ts-morph emit new indented statements.

## Deviations from Plan

**[Rule 3 - Blocking issue] BindingElement.remove() does not exist in ts-morph.**
The spec assumed `BindingElement.remove()`. ts-morph does not implement it for binding elements. Worked around by rebuilding the ObjectBindingPattern via `replaceWithText('{ a, b, c }')` after collecting the surviving element texts. Done last so descendant-forgetting doesn't invalidate the body reference used in D.3.

**[Rule 3 - Blocking issue] ts-morph default indent was 4 spaces.**
First test run failed only on indentation. Added `indentationText: IndentationText.TwoSpaces` to `manipulationSettings`. Existing fixtures unaffected.

**[Rule 3 - Reordering]** D.3 (body insert) executes BEFORE D.2 (destructure rebuild) because `replaceWithText` on a binding pattern forgets its descendants and ts-morph cannot guarantee the body node remains valid afterward in all cases. Ordered to keep things robust.
