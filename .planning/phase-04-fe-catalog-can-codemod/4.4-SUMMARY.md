## Deviations

### [Rule 3 - Blocking] Per-file build verification deferred within Task 1

The plan called for `npm run build` after each per-file codemod. In practice, processing `DocumentTable.tsx` first creates a transient TypeScript error window — `DatasetDetailPage.tsx` (file 5 in the strict order) still passes `isAdmin` as a prop to `<DocumentTable>`, so `tsc -b` fails with TS2322 until file 5 lands. The same parent/child receiver pattern applied to ConnectorListPanel→StandardTabRedesigned across the Task 1/Task 2 boundary. Resolution: ran lint per file (still useful as a syntactic gate, no errors), committed each file in strict order, and ran the full `npm run build` only at the end of each task. The first end-of-task-1 build still surfaced the StandardTabRedesigned consumer mismatch as expected — that resolved as soon as Task 2 landed StandardTabRedesigned.

### [Rule 1 - Bug] GlossaryPage.tsx unused `isAdmin` const

The codemod's Pattern B detected the multi-role OR chain in `GlossaryPage.tsx` and prepended a manual-migration TODO instead of rewriting. The local `isAdmin` const was never referenced in the JSX (the page only orchestrates two child tabs, both of which read permissions themselves), so the dead code blocked the build under `noUnusedLocals`. Removed the const plus the now-unused `useAuth` and `UserRole` imports. Commit `23bfc93`.

### [Rule 2 - Missing] Multi-role OR chain manual migrations (DatasetsPage / DatasetDetailPage)

Both files had `const isAdmin = user?.role === ADMIN || user?.role === LEADER`. The codemod's Pattern B is conservative and only handles single-role assignments — multi-role chains are intentionally left with a TODO marker (fixture 08 locks this behavior). However, Plan 4.4's verification grep is zero-tolerance for `user.role ===` matches, so leaving the TODOs would have failed acceptance. Manually migrated both to `useHasPermission(PERMISSION_KEYS.DATASETS_VIEW)`. Behavior is preserved because both `admin` and `leader` roles already hold `datasets.view` in the Phase 3 seed; the catalog gate is functionally equivalent to the OR chain. Commits `0c9d2e5`, `bebe4f5`.

### [Rule 2 - Missing] SystemToolsPage bare comparison

`SystemToolsPage.tsx:138` had a bare `user?.role === UserRole.ADMIN` JSX gate (not a `const isAdmin` assignment), which the codemod's Pattern C marked with a TODO above the enclosing component. Same rationale as above — manual migration to `useHasPermission(PERMISSION_KEYS.SYSTEM_TOOLS_VIEW)`, removed unused `useAuth`/`UserRole` imports. Commit `bc3318a`.
