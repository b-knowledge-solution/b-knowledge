  <action>
Append a new section to `fe/CLAUDE.md` titled `## Permission Gating: `<Can>` vs `useHasPermission` (Phase 4+)`.

The section MUST contain:

1. **The rule (one sentence each):**
   - Use `<Can I="action" a="Subject">` when you need CASL conditions — per-tenant, per-owner, per-instance reasoning. The check operates against a specific subject instance or class with conditions.
   - Use `useHasPermission(PERMISSION_KEYS.X)` for flat catalog-key feature gates with no per-instance reasoning. Pure boolean: "does this user have permission X?"

2. **A decision tree:**
   ```
   Need to gate a UI element on permission?
   ├── Does the check depend on a specific resource instance (this KB, this doc)?
   │   ├── YES → <Can I="update" a="KnowledgeBase" this={kb}>...</Can>
   │   └── NO  → continue
   ├── Does the check depend on subject CLASS without an instance (any KB)?
   │   ├── YES → <Can I="create" a="KnowledgeBase">...</Can>
   │   └── NO  → continue
   └── Flat capability check (e.g. "can submit feedback")?
       └── useHasPermission(PERMISSION_KEYS.FEEDBACK_SUBMIT)
   ```

3. **Examples:**
   ```tsx
   // Instance check
   <Can I="delete" a="KnowledgeBase" this={kb}>
     <Button onClick={() => deleteKb(kb.id)}>Delete</Button>
   </Can>

   // Flat capability check
   const canCreateDataset = useHasPermission(PERMISSION_KEYS.DATASET_CREATE)
   if (!canCreateDataset) return null
   ```

4. **Hard rules** (these EXACT phrases must appear in fe/CLAUDE.md so verification can grep them):
   - `Decision tree: <Can> vs useHasPermission` — section header
   - `isAdmin prop drilling is banned` — banned pattern
   - `Bare string keys are banned in useHasPermission — always use PERMISSION_KEYS.X` — no-hardcoded-strings rule
   - `Role-string comparisons (user.role === '...') are banned outside features/auth/` — covers P4.5 ESLint rule
   - The `PERMISSION_KEYS` const is auto-generated from `fe/src/generated/permissions-catalog.json`. If you need a key that doesn't exist yet, regenerate the snapshot from the BE: `npm run permissions:export-catalog` from the root.

5. **Where the pieces live:**
   - `fe/src/lib/ability.tsx` — `<Can>`, `useAppAbility()`, `AbilityContext`
   - `fe/src/lib/permissions.tsx` — `useHasPermission(key)`
   - `fe/src/constants/permission-keys.ts` — `PERMISSION_KEYS` (AUTO-GENERATED)
   - `fe/src/generated/permissions-catalog.json` — committed snapshot from BE

Place the section after the existing `## Key Rules` section. Preserve all existing content of `fe/CLAUDE.md`.
