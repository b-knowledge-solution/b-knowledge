# Phase 4 Plan 4.1: BE Catalog Exporter + FE Generator + useHasPermission Summary

Built the full snapshot-only catalog pipeline: BE exports the live permission registry to a committed JSON file, FE generator converts that into a typed `PERMISSION_KEYS` const, and a `useHasPermission` hook wraps the existing CASL ability with catalog-backed lookups — ready for P4.4/P4.5 codemod consumption.

## Tasks Completed

### Task 1 — BE permission catalog snapshot exporter
- **be/scripts/export-permissions-catalog.mjs** (92 lines): dynamic-imports the `@/shared/permissions/index.js` barrel under `tsx`, calls `getAllPermissions()`, validates (empty, dup keys, key-shape regex `[a-z0-9._-]`), sorts deterministically by `key`, and writes `fe/src/generated/permissions-catalog.json` with shape `{ generatedAt, permissions: [{ key, feature, action, subject, label, description? }] }`.
- **package.json:** added `"permissions:export-catalog": "cd be && tsx scripts/export-permissions-catalog.mjs"`.
- **fe/src/generated/permissions-catalog.json:** 112 keys, alphabetically sorted, no duplicates, all valid shape.
- Commit: `cdde661`

### Task 2 — FE generator + PERMISSION_KEYS const + npm hooks
- **fe/scripts/generate-permission-keys.mjs** (~95 lines): reads the BE snapshot, normalizes each key to SCREAMING_SNAKE (`.` and `-` → `_`, uppercase), hard-fails on duplicate normalized names, sorts by constant name, emits `fe/src/constants/permission-keys.ts` with AUTO-GENERATED banner, `export const PERMISSION_KEYS = { ... } as const`, and `export type PermissionKey = (typeof PERMISSION_KEYS)[keyof typeof PERMISSION_KEYS]`.
- **fe/src/constants/permission-keys.ts:** 112 entries (auto-generated).
- **fe/package.json:** added `permissions:generate-keys`, `predev`, `pretest`; composed existing `prebuild` to `rimraf dist && node scripts/generate-permission-keys.mjs`.
- Commit: `67f7e30`

### Task 3 — useHasPermission hook (TDD)
- **fe/src/lib/permissions.tsx** (~75 lines): module-scope `CATALOG_MAP: Map<string, {action, subject}>` built once from the imported JSON; `useHasPermission(key: PermissionKey): boolean` calls `useAppAbility()`, looks up `(action, subject)`, returns `ability.can(action, subject)`; unknown keys `console.warn` and return `false`. Full JSDoc with `@description`/`@param`/`@returns`/`@example`. No `useMemo`/`useCallback`/`React.memo`.
- **fe/tests/lib/permissions.test.tsx** (~90 lines): 4 Vitest cases — (1) grant with `read KnowledgeBase`, (2) deny when rule missing, (3) deny with empty ability, (4) `@ts-expect-error` bare-string case that also verifies runtime fallback returns `false`. Wraps `renderHook` with a fake `AbilityContext.Provider`.

## Deviations from Plan

### [Rule 3 — Blocker] Root script runs under `cd be`
- **Found during:** Task 1
- **Issue:** Plan specified `"permissions:export-catalog": "tsx be/scripts/export-permissions-catalog.mjs"` from root, but the permissions barrel uses `@/*` path aliases which tsx only resolves when the active tsconfig (`be/tsconfig.json`) is found via cwd.
- **Fix:** Changed script to `"cd be && tsx scripts/export-permissions-catalog.mjs"`. Functionally identical, just makes path aliasing work.

### [Rule 3 — Blocker] Auth-feature vi.mock added to permissions test
- **Found during:** Task 3 verification
- **Issue:** Importing `@/lib/ability` in the test file pulls in `@/features/auth` through `AbilityProvider`'s `useAuth` dependency. Without mocking it, Vitest's transform/collect phase hangs indefinitely (confirmed by reproducing and comparing to `tests/lib/ability.test.tsx`, which mocks the same module).
- **Fix:** Added `vi.mock('@/features/auth', () => ({ useAuth: () => ({ user: null }) }))` to the test file — mirrors the pattern already used in `ability.test.tsx`. Test suite then runs in ~15s.

### [Rule 2 — Plan acceptance criterion unfalsifiable]
- **Acceptance criterion:** "At least the `DATASET_VIEW: 'dataset.view'` line is present".
- **Reality:** The BE permission registry has no `dataset.*` keys — it uses `knowledge_base.*`, `agents.*`, etc. The plan author used an illustrative example that doesn't exist in this codebase.
- **Fix:** Tests use the real keys `KNOWLEDGE_BASE_VIEW` / `KNOWLEDGE_BASE_DELETE`. All other criteria (AUTO-GENERATED header, `as const`, `PermissionKey` type, tsc clean) are satisfied.

### Stray auto-commit `45c2eff`
- **Observation:** Between my Task 2 and Task 3 commits a commit `45c2eff feat(permissions): implement catalog-backed permission hook and tests` appeared that I did not author. It contained an earlier draft of `permissions.tsx` + `permissions.test.tsx` plus unrelated nginx and `vite.config.ts` edits.
- **Assessment:** Looks like a pre-commit hook or background automation snapshotted my unstaged edits mid-flow. The final state on disk is correct (my files supersede the stray commit) and all verification passes, so I left the history as-is rather than rewriting it. The stray commit's nginx/vite changes are unrelated to this plan and were not reverted (they look like pre-existing in-flight work in the working tree from earlier in the session).
