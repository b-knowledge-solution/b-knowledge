## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] exactOptionalPropertyTypes strict-mode violations**
- **Found during:** Task 2 build verification
- **Issue:** `Principal.sublabel` was always assigned `u.email` even when undefined; `buildCreateGrantBody`'s `kbId` parameter was passed as `number | null | undefined` from the editor handler. Both violate `exactOptionalPropertyTypes: true`.
- **Fix:** PrincipalPicker now conditionally spreads `sublabel` only when defined; ResourceGrantEditor's `handleAdd` passes `kbId: kbId ?? null` to the helper.
- **Files modified:** PrincipalPicker.tsx, ResourceGrantEditor.tsx
- **Commit:** `4bf2cff`

**2. [Rule 3 - Blocking] vitest.shared.ts allowlist gate**
- **Found during:** Task 1 first test run
- **Issue:** New test files must be added to `unitTestFiles` allowlist (documented pattern from 5.0b).
- **Fix:** Added both new test paths.
- **Commit:** `e2abcef`

**3. [Deviation] CategoryPicker out of scope for this plan**
- The plan sketch in Task 2 referenced `<CategoryPicker kbId={kb.id} value={selectedCategoryId} onChange={setSelectedCategoryId} />`. No such component exists, and building one is not in this plan's `files_modified` list.
- **Decision:** Render a hint message under the Category scope tab telling admins to open a category from the knowledge-base view to manage its grants. The wiring (`scope`, `selectedCategoryId`, `onScopeChange`) is in place so a future plan can drop a real picker in without further refactor.
- Documented as IOU below.

**4. [Deviation] PrincipalPicker tests use pure-helper strategy, not full render**
- The plan asked for behavior tests against a rendered component. Following the precedent set by `tests/features/datasets/ChangeParserDialog.test.tsx` (component rendering hangs in vitest due to babel-plugin-react-compiler + Radix/cmdk under jsdom), I extracted `buildPrincipalList` and `filterPrincipals` and tested those â€” they encode 100% of the filter chain. Same approach for `ResourceGrantEditor` via `buildCreateGrantBody`.
- All 5 specified behaviors are covered (one is split into 3a/3b for chip restrict + restore, totaling 6 tests in the picker file).
