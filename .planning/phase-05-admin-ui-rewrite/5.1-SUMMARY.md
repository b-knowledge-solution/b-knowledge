### Task 2 — PermissionMatrix component + PermissionManagementPage rewire (recovery commit)

**Component (`fe/src/features/permissions/components/PermissionMatrix.tsx`):**

- Registry-driven: rows sourced from `Object.values(PERMISSION_KEYS).sort()`, grouped by the dotted prefix via `groupPermissionKeys()`.
- Columns: `SUPER_ADMIN, ADMIN, LEADER, USER` from the canonical `UserRole` constants. Legacy `member`/`superadmin` intentionally excluded (Phase 6 removes them).
- Dirty state: `Map<role, Set<PermissionKey>>` tracking the target post-edit state per role.
- Sticky footer appears when `dirty.size > 0`:
  - **Cancel** → `setDirty(new Map())`, no API call, no confirmation dialog.
  - **Save (N)** → fires one `updateMutation.mutateAsync({ role, body: { permission_keys: [...keys] } })` per dirty role sequentially (for/await), then `globalMessage.success(sessionRefreshNotice)` and clears dirty state. On any failure: `globalMessage.error(saveError)` and the dirty state is preserved for retry.
- Every visible class has a `dark:` variant.
- Full JSDoc on the component, all four exported helpers, and the internal functions.

**Page (`fe/src/features/users/pages/PermissionManagementPage.tsx`):**

Thin wrapper — renders `<PermissionMatrix />` inside a padded container. The matrix owns its own card chrome and sticky footer.

### Task 3 — Tests (pure-helper, node env)

Rewrote the test file from a full-component jsdom render (which hung due to the shadcn/Radix + React Compiler interaction also observed by 5.2 and 5.3) into pure-helper tests running in node env. 13 tests across four helpers:

- `groupPermissionKeys`: 3 tests (grouping + ordering, no-dot edge case, empty input)
- `isCellChecked`: 3 tests (server fallback, dirty precedence, unknown role)
- `toggleInDirtyMap`: 4 tests (add, remove from server baseline, immutability, other-role preservation)
- `buildUpdatePayloads`: 3 tests (per-dirty-role payload, empty input, empty target array for full revoke)

All 13 pass in 4ms. No jsdom, no rendering, no hanging.

### Task 4 — i18n keys (handled by parallel agents)

The `permissions.admin.matrix.*` namespace keys (`title`, `permissionColumn`, `saveButton`, `cancelButton`, `saveError`, `sections.*`, `roles.*`) and the shared `permissions.admin.sessionRefreshNotice` were added to en/vi/ja.json as part of the cross-plan i18n coordination during Wave 1 — 5.2 and 5.3 both wrote i18n during their runs and the keys converged without conflict.

## IOUs

1. **Full-component jsdom test coverage** — the component's rendering path (especially Radix Checkbox and the sticky footer) is NOT directly tested. Pure-helper tests cover every *decision* the component makes (grouping, dirty state, server merging, payload shape) but do not exercise the DOM. If a regression is introduced purely in JSX/CSS wiring, it will escape unit tests. The Wave 2 plan 5.5 (Vitest suite) should add a manual UAT entry or a smoke playwright test for the rendered matrix.
2. **Section header i18n fallbacks** — `t(\`permissions.admin.matrix.sections.${section}\`, section)` uses the section name as the fallback. Translators may want explicit keys for every section prefix (`datasets`, `agents`, `knowledge_base`, etc.). Low-priority polish.
3. **Save button label pluralization** — `t('permissions.admin.matrix.saveButton', { count: dirtyCount })` relies on i18next plural forms in each locale. Verify the vi/ja plural forms render correctly during the 5.4 i18n audit.
