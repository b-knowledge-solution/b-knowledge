# Phase 5 Plan 5.2: User Detail Page + Permission Override Editor Summary

## One-liner
New `/iam/users/:id` admin detail page with tabbed Profile + Permissions surfaces, allow/deny override editor wired to the Wave 0 query hooks, and a client-side effective-permissions merge panel — unlocking per-user permission management entirely from the UI (D-03, D-04, D-05).

## What Was Built

### Task 1 — UserDetailPage shell, routing, row-click nav — commit `8befb05`
- **`UserDetailPage.tsx`**: React Router `:id` route component with shadcn Tabs, deep-linking via `?tab=` query param, loading spinner + not-found empty state, back-link to the user list.
- **`UserProfileTab.tsx`**: read-only dl-based display of `displayName`, `email`, `role`, `department`.
- **`useUser(userId)`** added to `userQueries.ts` — reuses the cached `users.list()` query via TanStack Query's `select:` option to pluck the matching user without any additional HTTP call.
- **`App.tsx`**: lazy import + `iam/users/:id` route wrapped with `<FeatureErrorBoundary><NavRoleGuard>`. Sidebar nav `getRoutePermission` already does longest-prefix matching, so `/iam/users/:id` resolves to `USERS_VIEW` automatically without any sidebarNav.ts edit.
- **`routeConfig.ts`**: added `/iam/users/:id` entry with `users.detail.title` i18n key.
- **`RoleManagementTable.tsx`**: rows now carry `onClick={handleRowNavigate}` with a `cursor-pointer hover:bg-slate-50 dark:hover:bg-slate-800` treatment. The actions `<TableCell>` stops click propagation so the inline role Select + dropdown menu continue to work normally.
- **`UserDetailPage.test.tsx`**: 3 jsdom tests covering the default-profile, `?tab=permissions` deep-link, and click-to-switch flows. Overrides the global `react-router-dom` mock in `tests/setup.ts` so real `useSearchParams` drives the URL state; mocks `OverrideEditor` to avoid pulling cmdk into the transform graph.

### Task 2 — OverrideEditor + PermissionKeyPicker + EffectivePermissionsPanel — commit `138610a`
- **`OverrideEditor.tsx`**: two `<Card>` sections (Allow overrides, Deny overrides) each with a catalog picker `[+ Add]` button and `[×]` remove buttons per row. Wires up Wave 0 hooks `useUserOverrides`, `useCreateOverride`, `useDeleteOverride`. Every successful mutation triggers `globalMessage.success(t('permissions.admin.sessionRefreshNotice'))` for R-10. Errors surface as `globalMessage.error`. Exports a pure `partitionOverrides(rows)` helper used both by the component and by the unit tests.
- **`PermissionKeyPicker.tsx`**: shadcn `<Popover>` + `<Command>` wrapping `Object.values(PERMISSION_KEYS)` grouped by dotted-prefix feature. Accepts an `excludeKeys` prop so the allow picker hides keys already in the allow list (same for deny). Selection dismisses the popover and calls `onSelect(key)`.
- **`EffectivePermissionsPanel.tsx`**: collapsed `<details>` rendering the merged effective permission set (role defaults + allow overrides − deny overrides, sorted). Uses `useRolePermissions(userRole)` to fetch role defaults. Exports a pure `mergeEffective` helper for unit testing. Per RESEARCH §6, the merge is fully client-side — no `whoCanDo` fan-out.
- **`OverrideEditor.test.tsx`**: 9 vitest specs in node env exercising `partitionOverrides` (allow/deny split, insertion-order preservation, empty case) and `mergeEffective` (role-only, allow adds, deny removes, deny-wins-over-allow, dedup, sort). Added to `vitest.shared.ts` `unitTestFiles` allowlist alongside PrincipalPicker and ResourceGrantEditor.

### Task 3 — i18n keys in 3 locales
- Keys added to `en.json`, `vi.json`, `ja.json` under `permissions.admin.sessionRefreshNotice`, `permissions.admin.overrides.*`, `permissions.admin.effective.title`, and `users.detail.*` (tabs, fields, title, notFound, backToList).

## Deviations from Plan

### Auto-fixed issues

**1. [Rule 3 — Blocking] Vitest collection hang when importing cmdk-rooted components**

- **Found during:** Task 2 first test run.
- **Issue:** `vitest run -c vitest.ui.config.ts OverrideEditor` hung indefinitely during the "collect" phase because any file that transitively imports `@/components/ui/command` pulls `cmdk` into the Vite transform graph, which interacts badly with `babel-plugin-react-compiler` under jsdom. `vi.mock('cmdk', ...)` and `vi.mock('@/components/ui/command', ...)` did not help because the mock only intercepts at the module-loader level — Vite still resolves and transforms the real file during graph walk.
- **Fix:** Followed the pattern already used by the sibling tests `PrincipalPicker.test.tsx` and `ResourceGrantEditor.test.tsx` (both headed by a NOTE comment explaining the cmdk/jsdom hang). Exported two pure helpers from the components — `partitionOverrides` from `OverrideEditor.tsx` and `mergeEffective` from `EffectivePermissionsPanel.tsx` — and rewrote the test to exercise those helpers directly in the `unit` (node) environment. 9 specs cover the same semantics the plan's original 6 behavioural tests targeted (allow/deny partition, allow adds, deny removes, deny-wins, dedup, sort).
- **Files modified:** `fe/src/features/permissions/components/OverrideEditor.tsx`, `fe/src/features/permissions/components/EffectivePermissionsPanel.tsx`, `fe/tests/features/permissions/OverrideEditor.test.tsx`, `fe/vitest.shared.ts` (allowlist entry).
- **Commit:** `138610a`

**2. [Rule 1 — Bug] `useSearchParams` returned empty values under the global router mock**

- **Found during:** Task 1 first test run.
- **Issue:** `tests/setup.ts` installs a process-wide `vi.mock('react-router-dom', …)` that replaces `useSearchParams` with a fixed empty `URLSearchParams`. Under that mock, my `?tab=permissions` assertion failed because the page always read `tabParam === null`.
- **Fix:** The new test file re-mocks `react-router-dom` with `vi.importActual` so real `MemoryRouter` + `useSearchParams` drive the URL state for these 3 specs only; other tests are unaffected.
- **Files modified:** `fe/tests/features/users/UserDetailPage.test.tsx`

**3. [Rule 1 — Bug] Radix Tabs did not activate under `fireEvent.click` in jsdom**

- **Found during:** Task 1 "switches tabs on click" spec.
- **Issue:** Radix Tabs listens on pointerdown/keydown rather than `click` in the current version, so a synthetic click didn't change the active tab.
- **Fix:** The click test dispatches `mouseDown` + `keyDown{Enter}` + `click` and then asserts the perms content (OverrideEditor stub) is rendered — avoiding brittle `data-state` attribute introspection.
