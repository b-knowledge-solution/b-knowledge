# Phase 5 Plan 5.6: Effective Access Page Summary

## One-liner
Ships the D-11 "Effective Access" admin page as a single-feature dropdown that fires one `whoCanDo` per selection and drills each user row into the P5.2 override editor — the final plan of Phase 5.

## What Was Built

### Task 1 — `EffectiveAccessPage` + tests — commit `dd03a9f`
- **`fe/src/features/permissions/pages/EffectiveAccessPage.tsx`** — new page component. Reads the current selection from `?key=…` (defaults to `knowledge_base.view`), decodes it with the `decodePermissionKey` pure helper into the `(action, subject)` pair expected by the BE whoCanDo endpoint, and calls `useWhoCanDo(action, subject)` exactly once per render. Renders:
  - A grouped `<select>` dropdown using `groupPermissionKeys` (reused from P5.1's PermissionMatrix) so options are organised by feature prefix (`agents`, `knowledge_base`, `users`, …).
  - A loading message while the query is pending.
  - An inline error message on query failure.
  - An empty-state message when the tenant has zero users with the selected permission.
  - A clickable users table (display_name, email, role, ✓) — row click calls `navigate(buildUserDetailUrl(user.id))` → `/iam/users/:id?tab=permissions`, landing directly on the P5.2 override editor tab.
- **Pure-helper exports** (tested in node env):
  - `decodePermissionKey(key)` — splits on the FIRST dot only (so `users.view_sessions` stays intact as action), PascalCases the snake_case prefix (`knowledge_base` → `KnowledgeBase`).
  - `buildUserDetailUrl(userId)` — returns the exact `/iam/users/:id?tab=permissions` deep-link shape the P5.2 UserDetailPage reads in its tab-sync effect.
- **Barrel** (`fe/src/features/permissions/index.ts`) — re-exports `EffectiveAccessPage` default + both pure helpers.
- **Tests** (`fe/tests/features/permissions/EffectiveAccessPage.test.tsx`) — 9 vitest specs in node env:
  - 7 tests for `decodePermissionKey` (simple, single-word, underscored suffix, multi-segment prefix, no-dot fallback, empty input, determinism).
  - 2 tests for `buildUserDetailUrl` (correct path, regex guard on the shape the P5.2 page depends on).
- **`vitest.shared.ts`** — added the test file to the `unitTestFiles` allowlist.

### Task 2 — Route mount + sidebar nav + i18n — commit `77bf269`
- **`App.tsx`** — lazy import for `EffectiveAccessPage` and `iam/effective-access` route wrapped in `<FeatureErrorBoundary><NavRoleGuard>`, following the exact pattern used by `iam/permissions`.
- **`routeConfig.ts`** — added `/iam/effective-access` metadata with `titleKey: 'permissions.admin.effectiveAccess.title'`.
- **`sidebarNav.ts`** — new IAM child entry gated by `PERMISSION_KEYS.PERMISSIONS_VIEW` (read-only scope), label `nav.effectiveAccess`, icon `Search` (already imported).
- **i18n in 3 locales** — added `nav.effectiveAccess` + `permissions.admin.effectiveAccess.{title,subtitle,selectFeature,user,email,role,has,empty,error}` to en.json, vi.json, and ja.json.

## IOUs (documented per plan frontmatter `must_haves.truths`)

### IOU #1 — Single-feature view instead of full cartesian matrix
The D-11 conversation originally imagined a features × principals cartesian matrix. Per RESEARCH §6 and §12, fanning `whoCanDo` across the ~112 permission keys in the catalog is O(keys × principals) and prohibitively expensive for v1 (hundreds to thousands of BE round-trips per page load). This plan ships the simpler single-feature dropdown view, which delivers the admin-useful slice ("who can do this ONE thing?") at O(1) backend call cost.

**Follow-up options if scale demands a richer view:**
- Server-side batch endpoint that returns a sparse matrix in one call.
- Client-side paginated / virtualized matrix with lazy per-cell queries.
- Snapshot-on-save caching so matrix cells don't re-query unchanged rows.

Pagination and virtualization for the users result list are also deferred — the current implementation renders the full list, which is fine for tenant sizes observed in production but will need windowing if a single feature ever returns thousands of users.

### IOU #2 — No team drill-down
`whoCanDo` in the BE service layer returns users only (verified in `be/src/modules/permissions/services/permissions.service.ts:439` — the service calls `ModelFactory.permission.whoCanDo` which returns `WhoCanDoEntry[]` shaped as `{ id, display_name, email, role }`). There is no team enumeration path in the current BE, so this page cannot show which teams have access without a new endpoint.

**Follow-up:** Extend `whoCanDo` (or add a sibling `whoCanDoTeams`) to include teams-with-access. Once that lands, this page can add a second "Teams" section below the users table, with row-click navigation into the P5.3 resource grant editor.
