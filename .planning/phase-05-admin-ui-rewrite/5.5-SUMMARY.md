# Phase 5 Plan 5.5: Vitest Coverage Thresholds Summary

Landed per-path coverage thresholds for Phase 5 permission surfaces. `src/lib/permissions.tsx` and `src/lib/ability.tsx` ship with real 85% enforcement; `src/features/permissions/**` ships with an honest 35% floor and a documented IOU to raise it once the shadcn/Radix jsdom hang is resolved.

## What was added

### `vitest.shared.ts` — global safety-net floor

Added a `coverage.thresholds` block with intentionally low numbers (lines 28, statements 28, functions 25, branches 18). This is NOT the real target — it's a regression tripwire. If someone deletes a test file wholesale, the global floor fails CI before the per-path thresholds even get a chance to evaluate.

### `vitest.ui.config.ts` — TS15 hook + `<Can>` at 85%

Per-path thresholds for the CASL primitives the whole FE depends on:

```ts
'src/lib/permissions.tsx': { lines: 85, statements: 85, functions: 85, branches: 80 },
'src/lib/ability.tsx':     { lines: 85, statements: 85, functions: 85, branches: 80 },
```

Both files currently measure at or near 100% via:
- `fe/tests/lib/permissions.test.tsx` (Phase 4 — useHasPermission hook)
- `fe/tests/lib/ability.test.tsx` (Phase 4 verified — `<Can>` rendering with mocked CASL rules)

11 tests total across the two files. The 85%/80% thresholds give headroom for small refactors while still failing CI on real coverage regressions.

### `vitest.unit.config.ts` — permission UI at enforceable 35%

```ts
'src/features/permissions/**': { lines: 35, statements: 35, functions: 25, branches: 25 },
```

**This is intentionally below the CONTEXT.md target of 85%.** The reason is documented inline in the config file and duplicated here so future-you doesn't re-raise it blindly:

The five component bodies (`PermissionMatrix.tsx`, `OverrideEditor.tsx`, `PrincipalPicker.tsx`, `ResourceGrantEditor.tsx`, `PermissionKeyPicker.tsx`) **cannot be mounted in jsdom**. Three independent Wave 1 agents (5.1, 5.2, 5.3) hit an indefinite hang caused by the interaction of shadcn/Radix primitives + `cmdk` + React Compiler + jsdom's DOM emulation. All three converged on the same workaround: extract testable logic into exported pure helpers (`groupPermissionKeys`, `toggleInDirtyMap`, `buildUpdatePayloads`, `partitionOverrides`, `mergeEffective`, `buildPrincipalList`, `filterPrincipals`, `buildCreateGrantBody`) and test those in node env.

The result: the decision logic of every component is thoroughly tested (198 passing tests across the phase) but the JSX bodies show up as uncovered in istanbul's report. The *behaviors* are covered; the *rendering paths* are not. Raising the threshold to 85% without fixing the jsdom hang would require either (a) running the test suite in a real browser via playwright (net-new infra), (b) diagnosing and patching the underlying jsdom/Radix/cmdk/RC interaction (unbounded investigation), or (c) adding no-op assertions that just mount-and-unmount components for coverage credit (anti-pattern).

The 35%/25% numbers are the actual current floors — if helpers get un-exported or tests get deleted, the threshold will fail CI.

## IOUs

1. **Raise `src/features/permissions/**` thresholds to 85%** — requires first solving the shadcn/Radix/cmdk/RC jsdom hang. Suggested approach: a dedicated investigation that adds a playwright smoke suite for the three admin pages (matrix, user detail, grant modal) and lets those drive coverage on the rendering paths. Track as a follow-up or SH (should-have) in the next milestone.
2. **Full-component smoke tests** — even without 85% coverage, a handful of shallow "this page renders without throwing" tests would catch import-level regressions. Currently gated on the jsdom hang fix. Same dependency as IOU #1.
3. **Coverage reporting to CI** — the thresholds will enforce locally via `npm run test:coverage`, but CI integration (fail the build on threshold breach) depends on the existing CI pipeline's test command. Verify that CI runs `npm run test:coverage` and not just `npm run test:run` — if it's the latter, the thresholds are local-only and don't actually guard main. Check `.github/workflows/` or equivalent during Phase 5 close.
