       .py->python, .js->javascript, .ts->typescript, .tsx->tsx, .jsx->tsx,
       .java->java, .go->go, .rs->rust, .rb->ruby, .cpp->cpp, .c->c,
       .cs->c_sharp, .php->php, .swift->swift, .kt->kotlin, .scala->scala,
       .lua->lua, .sh->bash, .bash->bash, .zsh->bash, .r->r, .dart->dart,
       .vue->vue, .svelte->svelte

    2. FUNCTION_NODE_TYPES dict: Map language -> list of AST node type strings for functions.
       python: function_definition, decorated_definition
       javascript/typescript/tsx: function_declaration, arrow_function, method_definition, function_expression
       java: method_declaration, constructor_declaration
       go: function_declaration, method_declaration
       rust: function_item
       (Add more as needed based on tree-sitter grammar node types)

    3. CLASS_NODE_TYPES dict: Similar mapping for class-level nodes.
       python: class_definition
       javascript/typescript/tsx: class_declaration
       java: class_declaration
       go: type_declaration
       rust: struct_item, impl_item

    4. _get_language_for_extension(ext: str) -> str | None

    5. _extract_imports(tree, source_bytes, language: str) -> str:
       Walk AST root children, find import_statement/import_from_statement (Python),
       import_declaration (JS/TS/Java), etc. Return concatenated import text as single string.

    6. _extract_scope_nodes(root_node, source_bytes, language: str) -> list[dict]:
       Recursively walk AST. For each function/class node, extract:
       - node_type: "function" or "class"
       - name: extracted from identifier child
       - signature: first line(s) up to body start
       - body: full node text
       - start_line, end_line
       - parent_class: name of enclosing class if method
       - decorators: text of decorator nodes if present
       - parameters: parameter list text
       - return_type: return type annotation if present
       - docstring: first string literal in body if present

    7. _estimate_tokens(text: str) -> int:
       Use len(text.split()) as rough token estimate (or use num_tokens_from_string if available).

    8. _split_large_function(node_dict: dict, max_tokens: int) -> list[dict]:
       If function body exceeds max_tokens, split at inner block boundaries (if/for/while/try blocks).
       Each sub-chunk gets parent function signature as content prefix.
       Return list of sub-node dicts.

    9. chunk(filename, binary=None, from_page=0, to_page=100000, lang="Chinese", callback=None, **kwargs):
       - Extract parser_config from kwargs
       - Get max_tokens from parser_config.get("chunk_token_num", 512)
       - Detect language from filename extension via _get_language_for_extension
       - If unsupported extension: fall back to naive text chunking (split by lines, group into chunks)
       - Parse source with tree-sitter: get_parser(language).parse(source_bytes)
       - Extract imports via _extract_imports
       - Extract scope nodes via _extract_scope_nodes
       - For each scope node:
         - If too large, split via _split_large_function
         - Build chunk dict with:
           content_with_weight = node text (with docstring inline)
           content_ltks = rag_tokenizer.tokenize(content)
           content_sm_ltks = rag_tokenizer.fine_grained_tokenize(content_ltks)
           docnm_kwd = filename
           title_tks = rag_tokenizer.tokenize(filename without extension)
           function_name = node name (metadata)
           class_name = parent class name (metadata)
           parameters = parameter text (metadata)
           return_type = return type text (metadata)
           decorators = decorator text (metadata)
           tag_kwd = ["code", language_name] (for filtering)
       - Store imports as metadata on first chunk or as a separate file-level chunk
       - Handle any code not inside functions/classes as a "module-level" chunk
       - Call callback with progress at regular intervals
       - Return list of chunk dicts

    IMPORTANT: Follow Google-style docstrings on all functions per CLAUDE.md conventions.
    IMPORTANT: Include inline comments above control flow, import handling, and AST traversal logic.
