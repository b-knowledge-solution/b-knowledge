1. Create `benchmarks/sample_conversations.py` with a `SAMPLE_CONVERSATIONS` list containing 10 multi-turn conversations (3-5 turns each) covering diverse domains:
   - Technical preferences ("I use TypeScript, prefer dark mode, use VS Code")
   - Project decisions ("We decided to use PostgreSQL instead of MongoDB")
   - Episodic events ("Yesterday's deployment failed because of the Redis connection")
   - Procedural knowledge ("To deploy, first run tests, then build, then push to Docker")
   - Contradictory updates ("Actually, we switched from PostgreSQL to CockroachDB" -- tests D-18 dedup)
   - Multi-fact messages ("I'm John, I work at Acme Corp, and I prefer Python")
   - Ambiguous content ("It was okay" -- tests extraction quality on vague input)
   - Long technical discussions (10+ sentences)
   - Short confirmations ("Yes, let's do that")
   - Mixed language hints (English with technical terms)
   Each conversation is a list of `{"role": str, "content": str}` dicts.

2. Create `benchmarks/test_extraction_quality.py` with these test functions:

   **`test_extraction_comparison`** (D-06):
   - For each of the first 5 SAMPLE_CONVERSATIONS:
     a. Run through mem0: `m.add(messages=conv, user_id="quality_test_user", infer=True)` -- capture extracted memories
     b. Document the extracted memories (content, count, event type)
   - Print a comparison table showing:
     | Conversation # | mem0 Memories Extracted | mem0 Content Samples |
   - The output IS the deliverable -- this feeds into the ADR qualitative section
   - Cleanup test index after

   **`test_deduplication_behavior`** (D-18):
   - Create a Memory instance with a persistent index (no cleanup between steps)
   - Step 1: Add conversation with "I prefer dark mode"
   - Step 2: Add conversation with "I like dark mode" (semantically identical)
   - Step 3: Add conversation with "I switched to light mode" (contradictory update)
   - After each step, call `m.get_all(user_id=...)` and print the full memory state
   - Assert: After step 2, memory count should NOT double (dedup worked)
   - Assert: After step 3, the preference should be updated to "light mode" (conflict resolution worked)
   - Print the event types returned by each `add()` call (ADD, UPDATE, DELETE, NOOP)
   - Cleanup test index

   **`test_memory_versioning`** (D-19):
   - Add a memory, update it (via contradictory add), then check history
   - Call `m.history(memory_id=...)` on the updated memory
   - Print the history entries showing old_memory -> new_memory transitions
   - Assert history has at least 2 entries (original ADD + UPDATE)
   - Cleanup test index

   **`test_custom_instructions`** (D-17):
   - Create a mem0 config with `custom_instructions`: "Only extract technical preferences and tool choices. Ignore personal information and greetings."
   - Add a conversation containing both technical and personal info: "Hi, I'm Alice. I'm 30 years old. I use Neovim and prefer Rust over C++."
   - Assert: Extracted memories include tool/language preferences but NOT personal details (age, greeting)
   - Print extracted memories for qualitative comparison
   - Cleanup test index

   **`test_forgetting_capability`** (D-13):
   - Add 5 memories to a pool
   - Call `m.delete(memory_id=...)` on one -- verify it's gone from `get_all()`
   - Call `m.delete_all(user_id=...)` -- verify all are gone
   - Document: mem0 has hard-delete only, no soft-delete/forgotten state. The wrapper must implement soft-delete if needed.
   - Cleanup test index

Each test should print results in a structured format (markdown table or clear labeled output) suitable for copy-paste into the ADR.
