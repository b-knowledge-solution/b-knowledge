<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> `/api/permissions/catalog` | Authenticated clients consume permission metadata; contract drift or overexposure would affect authorization UX. |
| permissions service -> Socket.IO clients | Server-originated mutation events fan out to connected browsers and must not become an untrusted client-write channel. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-1-01 | I | `GET /api/permissions/catalog` | mitigate | Keep the existing `requireAuth` + `requirePermission('permissions.view')` route gate and return only catalog metadata plus version, not tenant-specific grants or user-specific rules. |
| T-07-1-02 | D | permissions mutation event fan-out | mitigate | Emit only on successful permission mutations, not on reads or boot sync, so clients do not receive unnecessary invalidation storms. |
| T-07-1-03 | T | event-name / payload contract | mitigate | Centralize the event name in `be/src/shared/constants/socket-events.ts` and cover emitted payload shape in `be/tests/permissions/catalog-versioning.test.ts`. |
| T-07-1-04 | E | stale catalog after backend change | mitigate | Compute a deterministic version from registry contents and include it on both HTTP response and socket payload so FE can invalidate only when the server version changes. |
</threat_model>
