# Milestone v0.1: Document Upload Pipeline

**Status:** Shipped 2026-04-02
**Scope:** Document upload → converter → parse pipeline with full UI parity
**Commits:** 4bd80e8, b783c58, d6b07dc

## Overview

Closed the three critical gaps in the document upload pipeline: (1) per-upload parser selection, (2) the missing converter-queue.service.ts that writes converter jobs to Redis for the Python worker, and (3) auto-trigger parsing after upload. Then achieved full dataset-detail UI parity for version document management.

## Key Accomplishments

1. Created `converter-queue.service.ts` — the missing bridge between Node.js backend and Python converter worker via Redis job queue
2. Implemented auto-parse pipeline: non-Office files parse immediately on upload, Office files route through converter
3. Added converter job status endpoint and Force Convert Now button for immediate processing
4. Achieved full dataset-detail parity in version document list (18 features verified)
5. Fixed pre-existing bug: bulk operations were sending doc IDs instead of file names to project endpoints
6. Created 57 backend tests covering converter queue, pipeline flow, Redis key contracts, and parser selection

## Key Decisions

- Converter queue via Redis hashes (matches Python worker's existing polling model)
- Call RAG endpoints directly for version doc operations (avoid duplicating backend logic)
- Reuse dataset components via barrel exports (ProcessLogDialog, ChangeParserDialog, MetadataManageDialog)
- Office files skip auto-parse (need PDF conversion first)
